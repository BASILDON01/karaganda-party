# Подключение реальной оплаты (Kaspi / Halyk) — пошаговая инструкция

В проекте уже реализован флоу: создание заказа → редирект на оплату → вебхук → выдача билетов. Чтобы включить **реальные** платежи, нужно зарегистрироваться у платёжного агрегатора, получить ключи и прописать их в окружении.

---

## Варианты подключения и тарифы (кратко)

### ApiPay.kz (Kaspi Pay) — дорого для старта

- **Тарифы:** от 10 000 ₸/мес (до 30 платежей/день), далее 25 000–60 000 ₸/мес. Комиссия за транзакции: 0%.
- Подходит, если уже есть стабильный поток платежей. Для малого объёма фиксированная плата может быть высокой.

### Прямое подключение Kaspi Pay (партнёр Kaspi)

- **Комиссия:** 0,95% от платежа + 1 950 ₸/мес (со второго месяца). Подключение и первый месяц — бесплатно.
- Заявка: **kaspi.kz** → партнёрство / webpay. Нужны ИП/ТОО, ЭЦП, возможны требования к инфраструктуре (VPN и т.п.). Интеграция сложнее, чем через агрегатора.
- Может быть выгоднее ApiPay при небольшом количестве платежей.

### Halyk Bank ePayment — свой сайт, не их платформа

- **Важно:** создавать сайт на платформе Halyk **не обязательно**. Вы делаете сайт у себя (как factorkz.com), а Halyk даёт **API и платёжную страницу**.
- Схема: ваш сайт → по API создаёте платёж → пользователь переходит на **их** страницу оплаты (или открывается их виджет) → после оплаты Halyk шлёт вебхук на ваш сервер. Сайт остаётся ваш.
- **Комиссия:** от ~0,9%. Получение доступа: **halykid@halykbank.kz**, документация: **epayment.kz/docs**.
- В коде нужно добавить ветку под ePayment (OAuth → создание платежа → ссылка на оплату) в `payment-provider.ts` и при необходимости донастроить вебхук под их формат.

### Другие варианты

- **CloudPayments (Казахстан)** — приём карт Visa/Mastercard, Apple Pay, Google Pay. Условия и комиссии — по запросу. Сайт свой, интеграция через API.
- **Только переводы по номеру телефона / QR** — можно рассмотреть сервисы вроде «отправить ссылку на оплату» (если есть в РК), но для билетов удобнее единый эквайринг с вебхуком.

Итого: если ApiPay дорого — можно рассмотреть прямое партнёрство Kaspi (0,95% + 1 950 ₸/мес) или Halyk ePayment (свой сайт + их платёжная страница, комиссия от 0,9%). Свой сайт на «платформе» Halyk создавать не нужно.

## Что уже есть в коде

- **Тестовый режим** (сейчас): при нажатии «Оплатить» билеты создаются сразу, деньги не списываются.
- **Боевой режим**: при `PAYMENT_MODE=live` и наличии API-ключа создаётся заказ, вызывается API агрегатора, пользователь уходит на страницу оплаты; после успешной оплаты агрегатор вызывает ваш вебхук, по нему создаются билеты.

Вам нужно только: выбрать агрегатора, зарегистрироваться, получить ключ и настроить переменные окружения + вебхук.

## Вариант 1: Kaspi Pay через ApiPay.kz

### Шаг 1. Регистрация у агрегатора

1. Зайдите на сайт **https://apipay.kz** (или актуальный домен из их рекламы/поддержки).
2. Зарегистрируйте аккаунт (организацию/ИП/ТОО — как требуют).
3. Подключите **Kaspi Business** в роли «Кассир» — обычно это делается через поддержку ApiPay (WhatsApp/телеграм/почта). Уточните у них точную процедуру и сроки.

### Шаг 2. Получение API-ключа

1. Войдите в личный кабинет ApiPay.
2. Найдите раздел «API», «Интеграция» или «Настройки».
3. Создайте или скопируйте **API ключ** (X-API-Key или аналог). Сохраните его в надёжное место — он понадобится для сервера.

### Шаг 3. Настройка вебхука (обязательно)

1. В личном кабинете ApiPay найдите раздел «Вебхук», «Уведомления» или «Callback».
2. Укажите URL вебхука:
   ```text
   https://factorkz.com/api/payment/webhook
   ```
   (если домен другой — подставьте свой).
3. Сохраните. Без вебхука билеты после оплаты создаваться не будут.

### Шаг 4. Переменные окружения на сервере

На сервере, где крутится сайт (в папке проекта или в панели хостинга), создайте/дополните `.env.local`:

```env
# Включаем боевую оплату
PAYMENT_MODE=live

# Ключ из личного кабинета ApiPay
APIPAY_API_KEY=ваш_секретный_ключ_из_кабинета

# Базовый URL API (уточните в документации ApiPay!)
APIPAY_BASE_URL=https://api.apipay.kz

# URL сайта для ссылки «вернуться после оплаты»
NEXT_PUBLIC_APP_URL=https://factorkz.com
```

- `APIPAY_BASE_URL` — возьмите из официальной документации ApiPay (может быть другой домен или путь).
- После изменений перезапустите приложение (например: `pm2 restart factorkz-web`).

### Шаг 5. Проверка формата API ApiPay

Текущий код отправляет запрос создания счёта так:

- **Метод:** `POST {APIPAY_BASE_URL}/v1/invoices`
- **Заголовок:** `X-API-Key: ваш_ключ`
- **Тело (JSON):**  
  `amount` (число, тенге),  
  `external_order_id` (наш ID заказа),  
  `description` (текст),  
  `return_url` (куда вернуть пользователя после оплаты),  
  при необходимости `phone_number`.

Ожидаемый ответ: в теле JSON поле со ссылкой на оплату (`payment_url`, `url` или `link`) и идентификатор счёта (`id` или `invoice_id`).

Если у ApiPay другие имена полей или другой путь — откройте `src/lib/payment-provider.ts` и подстройте URL, имена полей в `body` и разбор ответа под их документацию.

### Шаг 6. Формат вебхука ApiPay

Наш обработчик вебхука ожидает в теле запроса что-то вроде:

- идентификатор заказа: `external_order_id`, `order_id` или `merchant_order_id`;
- статус: `status` или `payment_status` со значением вроде `paid` / `completed` / `success`;
- при необходимости: `id` или `invoice_id` счёта.

Если ApiPay присылает другие названия полей — откройте `src/app/api/payment/webhook/route.ts` и добавьте маппинг их полей на наш `orderId` и статус «оплачен».

### Шаг 7. Тестовый платёж

1. Убедитесь, что `PAYMENT_MODE=live`, ключ и вебхук настроены, приложение перезапущено.
2. На сайте выберите билеты и нажмите «Оплатить» (способ оплаты Kaspi).
3. Должен произойти редирект на страницу оплаты ApiPay/Kaspi.
4. Оплатите тестовой суммой (если агрегатор даёт тестовый режим).
5. После успешной оплаты вас вернёт на `https://factorkz.com/payment/success?orderId=...`, билеты должны появиться в «Мои билеты».

Если редиректа нет — проверьте логи сервера (ошибки от API в `payment-provider` логируются в консоль). Если редирект есть, но билеты не создаются — проверьте, что вебхук действительно вызывается и что в `webhook/route.ts` правильно читаются поля от ApiPay.

## Вариант 2: Halyk Bank (ePayment / Halyk ID)

Для приёма оплаты картами Halyk и Halyk ID используется платформа **ePayment** (epayment.kz).

### Шаг 1. Получение доступа

1. Напишите в Halyk: **halykid@halykbank.kz** с запросом на подключение приёма платежей (тест и/или прод).
2. Получите параметры: **TerminalID**, **ClientID**, **ClientSecret** (или аналог по их схеме).

### Шаг 2. Документация

- Изучите официальную документацию: **https://epayment.kz/docs/api** (или актуальный URL от банка).
- Обычно нужны: получение OAuth-токена, создание платежа, редирект пользователя на платёжную страницу, при необходимости — вебхук о статусе оплаты.

### Шаг 3. Доработка кода

Сейчас в коде один провайдер под ApiPay-подобный API (создание счёта одним запросом). ePayment чаще всего работает по схеме: OAuth → создание платежа → ссылка на оплату.

Нужно:

1. В **`src/lib/payment-provider.ts`** по `paymentMethod === 'halyk'` вызывать не текущий `createPaymentInvoice`, а отдельную функцию под ePayment:
   - запрос токена (ClientID + ClientSecret);
   - запрос создания платежа с суммой, описанием, return_url и нашим `orderId`;
   - из ответа взять ссылку на оплату и (если есть) ID платежа.
2. Вернуть `{ paymentUrl, invoiceId }` в том же формате, чтобы текущий флоу (редирект, сохранение заказа) не менять.
3. Если ePayment присылает вебхук — указать в их кабинете URL:  
   `https://factorkz.com/api/payment/webhook`  
   и при необходимости в **`src/app/api/payment/webhook/route.ts`** добавить разбор их формата (маппинг их полей на наш заказ и статус «оплачен»).

Конкретные имена полей, заголовков и URL берите из актуальной документации ePayment.

## Чек-лист перед запуском боевой оплаты

- [ ] Выбран агрегатор (ApiPay для Kaspi и/или ePayment для Halyk).
- [ ] Зарегистрирован аккаунт, подключён Kaspi Business / Halyk (по инструкции агрегатора).
- [ ] Получен API-ключ (и при необходимости TerminalID, ClientID, ClientSecret для Halyk).
- [ ] В кабинете агрегатора указан вебхук: `https://ваш-домен/api/payment/webhook`.
- [ ] В `.env.local` на сервере заданы: `PAYMENT_MODE=live`, ключи, `APIPAY_BASE_URL` (и при необходимости переменные для Halyk).
- [ ] Приложение перезапущено после изменения `.env.local`.
- [ ] Проведён тестовый платёж: редирект на оплату, успешный возврат на сайт, появление билетов в «Мои билеты».

## Где что править в коде

| Задача | Файл |
|--------|------|
| URL и тело запроса к API агрегатора, разбор ответа (ссылка на оплату, ID счёта) | `src/lib/payment-provider.ts` |
| Разбор тела вебхука (идентификатор заказа, статус оплаты) | `src/app/api/payment/webhook/route.ts` |
| Создание заказа и редирект пользователя | уже реализовано в `src/app/api/tickets/purchase/route.ts` и на странице пати |

После настройки агрегатора и переменных окружения реальная оплата должна заработать без смены логики создания билетов — достаточно подогнать запросы и вебхук под формат выбранного провайдера.
