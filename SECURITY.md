# Безопасность сайта FactorKZ

Краткий отчёт о проверке уязвимостей и рекомендациях (по состоянию на 09.03.2026).

---

## Что уже сделано хорошо

- **Авторизация через Telegram** — подпись данных проверяется по HMAC-SHA256 (hash), используется `timingSafeEqual` для сравнения. Срок действия `auth_date` ограничен. Пароли не хранятся.
- **Сессия (JWT)** — cookie с токеном: `httpOnly`, `sameSite: lax`, в production `secure`. Секрет из `AUTH_JWT_SECRET`. Срок жизни 30 дней.
- **Админка** — доступ только по списку Telegram ID из `ADMIN_TELEGRAM_IDS`. Все админские API проверяют `getSessionUser()` и `isAdmin(user.id)`.
- **Доступ организатора** — редактирование/просмотр пати только если `party.createdBy === user.id`. Чужие пати возвращают 404.
- **Нет XSS через контент** — пользовательский контент не вставляется через `dangerouslySetInnerHTML`. Использование `innerHTML` только для очистки контейнера виджета (пустая строка).
- **Валидация входных данных** — в API проверяются обязательные поля (submit party, purchase, city и т.д.). Данные хранятся в JSON-файлах, не в SQL — классический SQL-инъекции нет.

## Исправлено: вебхук оплаты

**Проблема:** endpoint `POST /api/payment/webhook` был доступен без проверки. Любой, кто знает URL, мог отправить запрос с `orderId` и `status=paid` и получить билеты без оплаты.

**Решение:** добавлена проверка секрета. Если в `.env` задана переменная `PAYMENT_WEBHOOK_SECRET`, вебхук принимает запрос только при совпадении секрета в заголовке `X-Webhook-Secret` (или `Authorization: Bearer <секрет>`). Если переменная не задана, вебхук работает по-старому (для обратной совместимости).

**Что сделать:** при подключении реальной оплаты задать на сервере `PAYMENT_WEBHOOK_SECRET` (длинная случайная строка) и настроить у агрегатора отправку этого секрета в заголовке при вызове вебхука (если агрегатор поддерживает). Если агрегатор присылает подпись (signature) в теле запроса — можно доработать проверку в `src/app/api/payment/webhook/route.ts` под их формат.

## Исправлено: доступ к заказу по orderId (IDOR)

**Проблема:** `GET /api/payment/order/[orderId]` возвращал данные заказа (статус, сумма) любому, кто подставил orderId в URL — проверка сессии не выполнялась. Можно было подбирать или угадывать чужие заказы.

**Решение:** перед отдачей данных проверяется сессия: заказ возвращается только владельцу заказа (`order.userId === user.id`) или админу. Иначе возвращается 404.

## Рекомендации

1. **Секреты** — `AUTH_JWT_SECRET`, `TELEGRAM_BOT_TOKEN`, `ADMIN_TELEGRAM_IDS`, `PAYMENT_WEBHOOK_SECRET`, ключи API оплаты хранить только в `.env` на сервере, не коммитить в git. В репозитории оставить `.env.example` без реальных значений.

2. **HTTPS** — в production сайт должен работать только по HTTPS. Проверить, что в настройках хостинга включён SSL и редирект с HTTP на HTTPS.

3. **Заголовки безопасности** — в проекте добавлен `src/middleware.ts`: выставляются `X-Content-Type-Options: nosniff`, `X-Frame-Options: SAMEORIGIN`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-XSS-Protection: 1; mode=block`.

4. **Ограничение частоты запросов (rate limit)** — для защиты от перебора и DDoS можно ограничить число запросов к критичным API (логин, покупка билетов, вебхук) по IP или по пользователю. Сейчас rate limit в коде не реализован — при росте нагрузки имеет смысл добавить (например, через middleware или прокси).

5. **Зависимости** — периодически запускать `npm audit` и обновлять пакеты с известными уязвимостями.

6. **Бэкапы** — папка `data/` (пользователи, заказы, билеты, пати) критична. Регулярно делать бэкапы и хранить их отдельно от сервера.

## Контакты при инциденте

При обнаружении уязвимости или взлома обратитесь к разработчику или владельцу проекта и опишите проблему. Критичные уязвимости желательно не публиковать до исправления.
